## 1. Motor d'àudio — 17 troballes confirmades

- [x] **A-01 · ALTA · esforç petit** — Pulse timing is anchored to main-thread message arrival, not the worklet's sample-accurate audio time
  - **Fitxer:** `libs/sound/timeline-processor.js` (línies 219-220)
  - **Problema:** The worklet detects pulses sample-accurately, but _emitPulse posts only {type:'pulse', step, interval} with no audio timestamp (timeline-processor.js:219-220). On the main thread, _handleClockMessage sets this._lastPulseTime = this._ctx.currentTime at the moment the message ARRIVES (index.js:1794-1799), and _stepTime() extrapolates every future scheduled sample from that anchor (index.js:1634-1655). Every pulse message re-anchors the whole grid to a value polluted by render-quantum granularity + MessagePort latency + main-thread jank, so all BufferSource onsets (and onSchedule/noteProvider melodic notes) inherit per-pulse jitter and a systematic lag relative to the worklet grid. The 'sample-accurate clock' is effectively quantized to main-thread message delivery.
  - **Fix proposat:** The AudioWorkletGlobalScope exposes the `currentTime` global. In process(), capture blockTime = currentTime at the top, and include the exact event time in the message: this.port.postMessage({ type:'pulse', step, interval, time: blockTime + i*this.secondsPerSample }) (pass i into _emitPulse). On the main thread use this._lastPulseTime = Number.isFinite(msg.time) ? msg.time : now. _stepTime() then extrapolates from a drift-free, jitter-free audio-clock anchor; the existing Math.max(ctx.currentTime, when) guard in _schedulePlayerStart already protects against anchors slightly in the past. NOTE: timeline-processor.js is timing-critical — change is additive (new message field), keeps the 1e-9 epsilons untouched, and must go through full test suite + diff approval per repo rules.
  - **Verificació adversària (confiança high):** libs/sound/timeline-processor.js:220 — _emitPulse posts {type:'pulse', step, interval} with no audio timestamp (AudioWorkletGlobalScope currentTime is available but unused). libs/sound/index.js:1794-1799 — _handleClockMessage sets this._lastPulseTime = this._ctx.currentTime at message arrival, re-anchored on EVERY pulse. index.js:1634-1655 — _stepTime extrapolates all scheduled times from that anchor; tick() (1684, 1690-1709, 1723-1748) derives every BufferSource onset, onSchedule call, and noteProvider note time from it. No other mechanism corrects the anchor (grep: _lastPulseTime only set at 1799, nulled in stop at 1431). The anchor carries render-quantum quantization (~2.7ms @48kHz), MessagePort delivery latency, and main-thread jank, so scheduled inter-onset spacing inherits per-pulse anchor jitter exactly as described. Systemic: affects all apps using TimelineAudio.

- [x] **A-03 · ALTA · esforç mitjà** — Cycle-subdivision and polyrhythmic-voice samples are scheduled reactively (after the event already happened) instead of via the lookahead
  - **Fitxer:** `libs/sound/index.js` (línies 1813-1849)
  - **Problema:** Base pulses are scheduled ahead in tick() with future `when` times, but 'cycle' and 'voice' sounds are scheduled inside _handleClockMessage when the worklet message arrives: cycleTime = this._scheduledTimes.get(this._lastStep) ?? (now + 0.001) (index.js:1826-1827) and the same pattern for voices (index.js:1845-1846). (now + 0.001)` fallback shows the gap was known.

Keep the worklet 'cycle'/'voice' messages exclusively for the onCycle/onVoice visual callbacks (unchanged contract). Same approach for the fractional-voice 'seleccionados' triggers.

- [ ] **A-04 · MITJANA · esforç petit** — Sample scheduling depends solely on a throttleable setInterval; first pulse is also late by up to one scheduler tick
  - **Fitxer:** `libs/sound/index.js` (línies 1666-1768)
  - **Problema:** New evidence beyond the documented 'setInterval causes jitter / use RAF' item: (1) tick() can schedule nothing until the first worklet 'pulse' message sets _lastAbsoluteStep (baseStep == null guard, index.js:1673-1674), and the message handler never invokes the scheduler, so the step-0 click waits for the next setInterval fire — late by up to _schedulerEverySec (10-30 ms) on top of message latency, which is concrete mechanics behind the documented 'primer play atropellat'. (2) In background tabs setInterval is clamped to >=1000 ms while the worklet keeps running and port messages keep arriving — scheduling starves and playback gaps out. The previously recommended RAF replacement would make (2) strictly worse, since RAF is fully paused in background tabs.
  - **Fix proposat:** Piggyback scheduling on the unthrottled MessagePort: store the tick closure (this._tickFn = tick) in _startScheduler and call this._tickFn() synchronously at the end of the 'pulse' branch of _handleClockMessage. This fills the lookahead window the instant each pulse anchor updates — fixing first-pulse latency and keeping audio scheduled in background tabs — while the existing setInterval remains as a low-rate safety net. Keep the per-device profile values (desktop/balanced/mobile) untouched.
  - **Verificació adversària (confiança high):** libs/sound/index.js:1289-1377 — play() resets _lastAbsoluteStep = null (1324), posts 'start' to the worklet (1367), then _startScheduler(). tick() early-returns while baseStep == null (1673-1674), and the 'pulse' branch of _handleClockMessage (1795-1812) never invokes the scheduler — grep confirms tick is referenced only at the setInterval registration (1767). So step-0 audio waits up to _schedulerEverySec (clamped 10-30ms via _adaptSchedulerInterval, 1789, or profile override) after the first pulse message, then _stepTime(0) returns the already-past anchor and index.js:1018 clamps it to ctx.currentTime → first click late. - [ ] **A-06 · MITJANA · esforç petit** — No AudioContext statechange handling: suspended/interrupted context leaves the engine in a zombie playing state

  - **Fitxer:** `libs/sound/index.js` (línies 487-489)

- [ ] **LA-02** — ensurePreferredSampleRateContext replaces Tone's AudioContext without closing the old one
  - **Fitxer:** `libs/sound/audio-context-helper.js` (línies 51-70)
  - **Problema:** When an existing Tone context runs at a rate other than 44100 (Firefox/Linux default 48 kHz), the helper creates a new context and calls Tone.setContext(next), but never closes the replaced context — it keeps its audio device thread alive for the page lifetime.
